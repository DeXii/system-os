## Резюме

Проект имеет зрелую модульную структуру и продуманный DIRECTOR (Zod + allowlist). Критические риски сосредоточены в **cloud sync**, **day bootstrap**, **AI apply** и **рассогласовании метрик readiness/stage gates**. Ниже — находки по модулям; исправления Wave 1–3 отмечены как **FIXED** где применены в этом коммите.

## COMMAND

### [CRITICAL] Double-bootstrap → дубликаты missions — **FIXED** (single-flight)

**Проблема:** `generateFullStackMissions` / `generateFullStackProtocol` — check-then-act без транзакции.  
**Где:** `src/core/engines/mission-generator.ts`, `src/core/kernel/commands/bootstrap.ts`  
**Исправление:** `ensureDayBootstrapped` — single-flight lock per `date`.

### [HIGH] `dayOverrides` last-writer race — **FIXED**

**Проблема:** Параллельно `buildDay` и `buildTodayQueue` → phantom slots.  
**Где:** `src/core/engines/week-schedule.ts`, `CommandModule.tsx`  
**Исправление:** `runWithDayScheduleLock` на все мутации `dayOverrides` + merge по `taskKey`.

### [MEDIUM] Toggle mission/protocol без sync schedule — **FIXED**

**Проблема:** `toggleMission` обновляет только DB, не слот.  
**Где:** `src/modules/command/CommandModule.tsx`  
**Исправление:** При `done` вызывать `completeByTaskKey`.

### [MEDIUM] Stale `dayReports.compliance`

**Проблема:** `getOrCreateDayReport` не пересчитывает compliance.  
**Где:** `src/core/engines/command-compliance.ts`  
**Исправление:** Вызывать `refreshDayReportCompliance` после любого completion path.

### [MEDIUM] Palette briefing без refresh COMMAND — **FIXED**

**Где:** `src/app/OsLayout.tsx`, `director-service.ts` — `refresh()` + `emitOsRefresh` после command-scope tasks.
